An honest teardown

What is textpour, really?

textpour adds nothing Pretext can't be driven to do — it is built on Pretext's per-line layoutNextLine. The line-flow loop is about twelve trivial lines you could inline. The only reusable part is turning a 2-D shape into correct per-row spans. Proof below.

const prepared = prepareWithSegments(TEXT, FONT);
let cur = { segmentIndex: 0, graphemeIndex: 0 };

ctx.font = FONT;
ctx.textBaseline = 'alphabetic';
ctx.fillStyle = '#17130d';

for (let y = CY - R; y + LH <= CY + R; y += LH) {
  const yc = y + LH / 2, dy = yc - CY;
  if (Math.abs(dy) > R) continue;
  const half = Math.sqrt(R * R - dy * dy); // <-- you own this
  const x0 = CX - half, x1 = CX + half;
  const line = layoutNextLine(prepared, cur, x1 - x0);
  if (!line) break;
  ctx.fillText(line.text, x0, y + ASC);
  cur = line.end;
}

A glyph: the library earns it.

Swap the circle for a letter. The loop does not change. The spans function does.

// The 12-line loop is IDENTICAL. Only spansAt changed.
// You now maintain a rasterizer, a run-length scanner,
// and a font-size binary search yourself.

// 1. Rasterize the glyph to an offscreen canvas:
const mask = document.createElement('canvas');
mask.width = W; mask.height = H;
const mctx = mask.getContext('2d', { willReadFrequently: true });
let lo = 10, hi = 900;
for (let i = 0; i < 22 && hi - lo > 0.5; i++) {
  const s = (lo + hi) / 2;
  mctx.font = `900 ${s}px Fraunces`;
  const m = mctx.measureText('&');
  (m.width <= W - 80 && (m.actualBoundingBoxAscent
    + m.actualBoundingBoxDescent) <= H - 72) ? lo = s : hi = s;
}
mctx.font = `900 ${lo}px Fraunces`;
mctx.textAlign = 'center'; mctx.textBaseline = 'middle';
mctx.fillText('&', W / 2, H / 2);

// 2. Read alpha into a flat array:
const img = mctx.getImageData(0, 0, W, H).data;
const alpha = new Uint8ClampedArray(W * H);
for (let p = 0; p < W * H; p++)
  alpha[p] = img[p * 4 + 3] >= 128 ? 255 : 0;

// 3. Per-row run-length scan to find ink spans:
function spansAt(y) {
  const row = Math.floor(y);
  if (row < 0 || row >= H) return [];
  const spans = [];
  let inside = false, start = 0;
  for (let x = 0; x < W; x++) {
    const ink = alpha[row * W + x] > 0;
    if (!inside && ink) { inside = true; start = x; }
    else if (inside && !ink) { spans.push([start, x]); inside = false; }
  }
  if (inside) spans.push([start, W]);
  return spans;
}

// 4. Binary-search font size so words pack the shape:
// ... another ~20 lines to auto-fit body font size ...

// 5. Then the same 12-line loop, calling spansAt(y)
// for each row and threading cursor across all spans.
// You also re-derived justification slack distribution.

The loop never changed. spansAt exploded into a rasterizer, and you re-derived auto-fit too. That pile — for circles, donuts, concave polygons, SVG paths, glyphs, masks, and boolean combinations — is all textpour actually is.

textpour ≈ Region.spansAt(y) for arbitrary shapes + a twelve-line loop you could inline. One simple shape: inline it. Many or hard shapes — three-plus spans per row, glyphs, photo masks, booleans, justification slack across spans, emoji-correct carets — reuse it. It is a convenience kernel, not a new capability. That is the honest pitch.
